# Narrative: Core Split — Phase 2 (June 28, 2026)

## "engress gets its own brain"

The edge box was doing everything. Public tunnels, API, auth, metrics — all on one t4g.micro. Time to split.

## What happened

### The Terraform state was lost

`core/deploy/terraform/terraform.tfvars` had no state file (`terraform state list` → "No state file was found!"). Every resource — ECR, EC2, IAM, security groups, OIDC, CloudFront — was orphaned from Terraform. Applying would've created duplicates and conflicted with the live EIP.

**Fix:** Imported all 22 resources one-by-one (`terraform import ...`) — repos, IAM roles, instance profile, security groups, EC2 instance, EIP association, OAC, CloudFront function, OIDC provider. Set `spa_bucket_name` in tfvars to prevent the S3 rename that `name_prefix = "engress"` would've triggered.

### user-data ran on EC2, then broke

`workspace.sh` hardcoded `export AWS_PROFILE=ghostweasel-flux`. On the operator's machine this is correct. On the EC2 instance it meant every `aws` CLI call inside user-data died with "The config profile (ghostweasel-flux) could not be found, password is empty" — even though the instance had a perfectly good IAM role.

**Fix:** Detect EC2 via DMI UUID (`/sys/class/dmi/id/product_uuid` starts with `ec2`). Skip the profile when running on an instance. This worked because the SSM Run Command network namespace can't reach IMDS, but DMI is always readable.

### The edge instance got replaced

The Terraform apply randomly recreated the edge EC2 because `user_data` hash changed. The old `i-0df9b4c6fc758210b` got terminated; the original EIP `18.216.236.251` floated free for ~5 minutes before being reassociated to the new instance `i-0eb8c5238a07d810a`. DNS was down during that window.

### Session key mystery

`docker-compose.control.ecr.yml` was missing `FLUX_SESSION_KEY` in its environment block. The api container crashed immediately with "session_key must be >= 32 bytes (ENGRESS_SESSION_KEY)". The session key lives in `edge.yaml` (generated by `flux_write_edge_config` with `openssl rand -base64 32`), but the api compose never read it.

**Fix:** Added `FLUX_SESSION_KEY: ${FLUX_SESSION_KEY:-}` to the control compose environment. Also created `/opt/engress/core/deploy/data/api.yaml` with the session key and stored it in SSM (`engress-session-key`).

### Git remotes without credentials

`clone-private.sh` clones with a PAT from SSM, then resets the remote to the public URL. Subsequent `git pull` fails with "could not read Username for 'https://github.com'". The scripts repo was also stuck at an old commit (8326ab5) because the Phase 1 changes never got pulled on the instances.

## Blockers discovered

### Amplify deploy keys disabled

Creating the Amplify app fails with:
```
BadRequestException: Deploy keys are disabled for this repository
```

GitHub org/repo settings prevent Amplify from auto-provisioning a deploy key on `engress-io/core`. This blocks automatic SPA hosting on Amplify until an org admin re-enables deploy keys.

**Needed action:** Go to https://github.com/organizations/engress-io/settings/oauth_application_policy and enable deploy keys, or add the deploy key manually at https://github.com/engress-io/core/settings/keys after creating the Amplify app.

## Lessons learned

- **Never lose Terraform state.** Set up an S3 backend with DynamoDB locking before the next deploy. One accidental state loss cost hours of import work.
- **Profile vs IAM role matters.** Code that works on the operator's machine breaks on EC2 in non-obvious ways. Always detect the execution environment.
- **Session config must be explicit.** The edge generates a session key; the api must receive it. Implicit env var inheritance is fragile.
- **SSM Run Command has network constraints.** IMDS unreachable inside SSM, which broke the first attempt at EC2 detection via metadata probe. DMI filesystem reads work.
